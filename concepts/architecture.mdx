## What lives where

| Layer | What it does | Where it runs |
|---|---|---|
| **`@fibo/react-sdk`** | Renders auth UI, exposes hooks (`useFiboWallet`, `useFiboSwap`, etc.), passes user signatures from Privy back to Fibo | Browser, inside your React app |
| **Privy embedded wallets** | Holds the user's private keys in a hardware-backed TEE, gates signing on passkey / 2FA | Privy's infrastructure + the user's device |
| **Fibo backend** | Quoting, routing, transaction construction, gas sponsorship, KYC orchestration, ramping connectors, webhooks | Fibo-hosted (Node.js + Postgres) |
| **Your server** | Receives webhooks, calls privileged endpoints via `sk_…`, holds your business logic | Your infrastructure |

## Why the SDK is intentionally thin

The SDK doesn't know how to build a swap route, how to construct an MEE super-transaction, or which bridge to pick for an Ethereum → Solana transfer. All of that lives in the Fibo backend.

This isn't an oversight — it's a deliberate property:

- **You don't run a node, an indexer, or a transaction builder.** Your team writes product code, not protocol code.
- **Vendor changes are absorbed server-side.** When Biconomy ships v2 of their SDK, or LiFi changes a route format, you don't redeploy your frontend. We deploy the backend and your integration keeps working.
- **Security boundary is clear.** The SDK only ever holds your `pk_…` key. Anything that could move funds or read sensitive data requires `sk_…` and runs on your server.
- **Multi-platform parity is free.** Today: React web. Next: React Native. Same backend, same APIs.

## How a transaction actually flows

A swap is the canonical case — every transaction kind (yield deposit, transfer, buy_btc) follows the same shape:

<Steps>
  <Step title="Quote">
    The SDK calls `POST /v1/quote` with the swap parameters. Fibo backend asks LiFi for the best route, computes Fibo's fee, and returns a quote with a 5-minute TTL.
  </Step>
  <Step title="Prepare">
    The user clicks "Swap". The SDK calls `POST /v1/prepare-tx`. The backend builds the instructions (allowances + swap calldata + chain routing) and returns the signable payload — for EVM gasless flows this is an EIP-7702 authorization the user must sign once.
  </Step>
  <Step title="Sign">
    The SDK passes the payload to Privy. The user authenticates (passkey on web, biometrics on mobile) and Privy returns the signature. The key never leaves the TEE.
  </Step>
  <Step title="Submit">
    The SDK calls `POST /v1/submit` with the signature. Backend fetches the MEE quote, returns its hash for a second signature, then executes via Biconomy MEE — gas is paid from Fibo's gas tank.
  </Step>
  <Step title="Webhook">
    Backend tracks confirmation. When the on-chain bridge / swap completes, your server receives a `tx.confirmed` webhook with the `super_hash`, tracker URL, and final amounts.
  </Step>
</Steps>

This shape — quote / prepare / sign / submit / webhook — is uniform across all transaction kinds. Solana flows substitute the EIP-7702 step for a backend fee-payer co-signature; Bitcoin flows substitute PSBT signing via Privy's `useSignRawHash`. The user experience is the same: one or two biometric prompts and the transaction goes through gasless.

## What this means for you

- **You don't need crypto expertise on your team to integrate.** The SDK and webhooks expose a small, normalized surface — addresses, balances, transaction states, USD-denominated amounts. Bridge selection, gas optimization, chain abstraction, all happen behind the API.
- **You're not on the contractual chain of any provider.** Privy / Biconomy / LiFi contracts are signed by Fibo. Monerium / Transak / Sumsub contracts (the ones with regulatory weight) you sign yourself when you opt into ramping — Fibo provides the integration glue, not the legal relationship. See the [ramping guide](/guides/ramping).
- **You can switch off Fibo at any time without users losing access to their funds.** Because keys live in Privy's TEE under the user's control (not Fibo's), an end-of-life scenario is recoverable: users export their wallets via Privy's self-service flow and re-import elsewhere. There's no "Fibo holds your crypto" lock-in.
