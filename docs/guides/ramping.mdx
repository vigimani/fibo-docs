## The contractual model

Ramping (fiat → crypto and back) involves regulated entities and contracts that carry specific obligations: KYC, transaction monitoring, sanctions screening, AML reporting. Some of these obligations attach to whoever "operates" the ramping flow.

Fibo's position is intentional: **we don't operate ramping**. We facilitate it. The contracts that have regulatory weight (with Monerium, Transak, Sumsub) are signed by **you, the integrator**. Fibo provides the technical glue — the UI components, the share token bridge, the webhook plumbing — but is not on the contractual chain.

This is the Stripe Connect model applied to crypto ramping: Stripe doesn't accept payments on the merchant's behalf — the merchant accepts them, Stripe provides the infrastructure. Same here: you operate the ramping flow with the provider; Fibo provides the SDK and the data fabric.

The practical consequence: when you opt into Monerium, you sign a distributor agreement directly with Monerium. When you opt into Transak, you sign Transak's terms directly. Fibo never sees the user's IBAN, card PAN, or bank credentials — those flow between the user, the provider, and (where applicable) your servers.

## Two providers available today

### Monerium — EUR ↔ EURe (SEPA, EEA)

[Monerium](https://monerium.com) is an EMI (electronic money institution) that issues EURe, a 1:1 EUR-pegged stablecoin on Polygon and Ethereum. SEPA-only — works for users in any SEPA country with an EU bank.

Geographic limits: EEA + SEPA-eligible neighbors (Switzerland, Norway, etc.).
Settlement: typically instant (SEPA Instant) for EUR → EURe; standard SEPA (D+1) for EURe → EUR.
KYC: Monerium runs its own KYC, **but** if your users have already KYC'd via Sumsub (across your other products), Monerium accepts a [Sumsub share token](https://docs.sumsub.com/share-applicant-data) directly via their `POST /profiles/{id}/share` endpoint. Zero double-KYC for users.

### Transak — Card / bank → 100+ tokens, globally

[Transak](https://transak.com) is a multi-currency on-ramp that accepts card, bank transfer, Apple Pay, Google Pay, and SEPA. Coverage is global with the usual exclusions (sanctioned jurisdictions).

Geographic reach: 150+ countries with varying payment method support.
Settlement: card / Apple Pay typically minutes; bank transfer 1-3 days.
KYC: Transak runs its full KYC flow inline.

## What Fibo provides (baseline, all tiers)

Even if you build your own ramping flow with a different provider, four Fibo capabilities help you integrate:

1. **`wallet.funded` webhook** — Fibo's indexer detects incoming USDC / EURe / native transfers to your users' addresses and fires a webhook. Use this to unlock features once the user has funds, regardless of where the funds came from.
2. **`GET /v1/me/addresses`** — canonical endpoint returning the user's EVM / Solana / Bitcoin addresses, for feeding into any provider's "send to this address" flow.
3. **`useFiboBalance()` hook** — React helper returning live balance + a `hasFunds` boolean. Gate "Swap" / "Yield" CTAs on this.
4. **Integration patterns** — guides on this site cover how to integrate Monerium, Transak, Stripe Onramp, Mercuryo, and others alongside Fibo. The pattern is uniform; only the provider SDK differs.

These four work for **all** customers — no opt-in needed, no extra contract.

## What the opt-in connectors add

If you opt into a Fibo-built connector (`@fibo/react-sdk/monerium`, `@fibo/react-sdk/transak`), you get:

- **Drop-in React components** — `<MoneriumOnboardingCard />`, `<TransakWidget />`, etc. — that handle the provider's UI inside your app.
- **The Sumsub→Monerium share token bridge** (Monerium connector only) — if you already use Sumsub for KYC, Fibo's backend mints the share token and passes it to Monerium so your users skip the second KYC.
- **Webhook normalization** — Monerium's `order.completed` and Transak's `ORDER_COMPLETED` become a uniform `ramp.completed` Fibo event with the same shape across both.
- **Cross-provider routing (planned)** — pick the cheapest provider for a given user / amount / currency. Single API call, Fibo selects.

Each connector is a sub-export. Unused connectors are tree-shaken out:

```tsx
// Add Monerium SEPA → EURe to your app
import { MoneriumConnector, MoneriumOnboardingCard } from '@fibo/react-sdk/monerium';

function FundingPage() {
  return (
    <MoneriumConnector
      // Your Monerium client id — issued when you sign the distributor agreement.
      clientId={process.env.NEXT_PUBLIC_MONERIUM_CLIENT_ID!}
    >
      <MoneriumOnboardingCard
        onCompleted={() => router.push('/swap')}
      />
    </MoneriumConnector>
  );
}
```

## When to opt in vs roll your own

**Opt in to a Fibo connector** when:
- You don't have an existing ramping integration.
- You want the share token bridge (Sumsub → Monerium).
- You want webhook normalization across providers.
- You're OK signing the provider's distributor agreement.

**Roll your own** (using only the baseline four levers) when:
- You already have a deep integration with another ramping provider.
- You're in a jurisdiction Fibo's chosen providers don't cover.
- You want full control over the UI / branding of the funding flow.

Either way, the post-funding crypto flow (swaps, yield, transfers) is identical — funds land in the user's embedded wallet, your `wallet.funded` webhook fires, and the rest of the Fibo SDK works.

## A note on regulatory positioning

Fibo's view: non-custodial crypto infrastructure (wallets, gasless transactions, swaps, yield) is the part where we believe we're outside the CASP definition under MiCA — keys live in the user's TEE, we never hold funds, we don't intermediate fiat. Ramping is the part where regulatory weight is real, which is exactly why we structured it as a connector with you on the contractual chain rather than something Fibo operates.

This structure is mature for Stripe (payments) and for Twilio (telecom); applying it to crypto ramping is the same pattern.

For your own regulatory positioning, consult a lawyer in your jurisdiction — non-custodial software distribution alongside an opt-in ramping integration is a defensible posture in the EEA and most US states, but every jurisdiction has its quirks.
