## What webhooks do

Fibo emits webhooks for every asynchronous state change your integration needs to react to:

- A wallet was created or funded
- A transaction confirmed (or failed, or expired)
- A KYC status changed
- An API key was rotated
- A tenant-level alert fired (gas tank low, etc.)

Each webhook is one HTTP POST to the endpoint you configure, signed with HMAC-SHA256 (see [signatures](/docs/authentication/webhook-signatures)).

## Configuring your endpoint

You register one webhook URL per tenant per environment. Both `test` and `live` events go to the same URL by default — branch on the `livemode` field in the payload if you need to handle them differently. If you want hard separation, register two URLs (one with the `test` filter, one with `live`).

```http
POST /v1/webhook-endpoints HTTP/1.1
Authorization: Bearer sk_live_…
Content-Type: application/json

{
  "url": "https://your-server.example.com/fibo/webhooks",
  "enabled_events": ["*"]
}
```

`enabled_events: ["*"]` subscribes to everything; you can scope to a subset:

```json
{
  "enabled_events": ["wallet.*", "tx.confirmed", "tx.failed"]
}
```

## Delivery semantics

| Property | Behavior |
|---|---|
| Method | POST with `application/json` body |
| TLS | Required. We refuse to deliver to `http://` endpoints. |
| Timeout | 10 seconds per attempt |
| Success | Any 2xx response (we recommend 204 No Content) |
| Failure | Anything else — non-2xx, timeout, TLS error, DNS error |
| Retry | Exponential backoff: 30s, 5min, 30min, 2h, 12h, 24h, 24h. After 7 failed attempts the event is marked `abandoned` and stops retrying — it remains queryable via `GET /v1/webhook-events`. |
| Ordering | Per-resource ordering preserved on the happy path. Under retry pressure (e.g., your endpoint is down for 30 minutes), Fibo may deliver `tx.confirmed` for transaction B before `tx.failed` for transaction A. Don't rely on global ordering; idempotency-keyed processing is the correct model. |
| Idempotency | Each delivery includes a unique `event_id`. Replays use the same `event_id` — store it and skip duplicates. |

## Anatomy of an event

```json
{
  "event_id":   "evt_2k4j…",
  "event_type": "tx.confirmed",
  "livemode":   false,
  "created_at": "2026-05-20T10:32:00.123Z",
  "data": {
    "transaction_id": "tx_8x9w…",
    "user_id":        "usr_…",
    "kind":           "swap",
    "status":         "confirmed",
    "super_hash":     "0x…",
    "tracker_url":    "https://meescan.biconomy.io/details/…",
    "source":         { "amount": "100.0", "currency": "USDC", "chain_id": 8453 },
    "destination":    { "amount": "0.0312", "currency": "ETH",  "chain_id": 8453 },
    "fees":           { "fibo": "0.50", "currency": "USDC" }
  }
}
```

Fields you can rely on across all event types:

- `event_id` — UUIDv7, monotonically increasing. Idempotency key.
- `event_type` — dot-namespaced (`<resource>.<verb>`). See the [events catalog](/docs/webhooks/events).
- `livemode` — `true` for live keys, `false` for test keys.
- `created_at` — ISO-8601 UTC.
- `data` — the resource payload. Shape varies by `event_type`.

Future fields may be added to `data` without warning. **Treat the payload as schema-loose** — destructure what you need, ignore the rest.

## Handler patterns

The handler shape that works under all conditions Fibo can throw at it:

```ts
app.post('/fibo/webhooks', express.raw({ type: 'application/json' }), async (req, res) => {
  // 1. Verify signature (drop unsigned in 401)
  // 2. Parse
  const event = JSON.parse(req.body.toString('utf8'));

  // 3. Idempotency check — store event_id, return 204 if seen before
  if (await alreadyProcessed(event.event_id)) {
    return res.sendStatus(204);
  }

  // 4. Acknowledge FIRST, process in the background
  res.sendStatus(204);

  // 5. Process — write to DB, fan out to your business logic
  await processEvent(event).catch((err) => {
    log.error({ err, event_id: event.event_id }, 'fibo webhook handler failed');
    // Throwing here doesn't help — we already returned 204. Surface the
    // failure to your own retry/observability stack.
  });
});
```

Why ack-then-process: if your handler takes 9 seconds to write to DB and then your DB is down, you'll time out on Fibo's side and we'll retry. The retry replays the event — your DB recovers in the meantime — but you've now incremented your "events received" counter twice without doing anything useful. Idempotency + early ack short-circuits this.

## Replay & backfill

If your endpoint was down and an event was abandoned (7 retries exceeded), or you need to re-process events for any reason, use the dashboard's replay UI or `POST /v1/webhook-events/{id}/replay`. Replays carry the same `event_id` as the original — your idempotency check will skip them if you've already processed them; otherwise they execute as normal.

Backfilling on first integration: `GET /v1/webhook-events?from=2026-01-01` returns all historic events for your tenant. You can iterate and `POST /v1/webhook-events/{id}/replay` selectively.
