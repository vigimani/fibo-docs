---
title: Rotate the signing secret
description: POST /v1/webhook-endpoints/{id}/rotate-secret — mint a new HMAC signing secret with a brief overlap window.
sidebar_position: 4
---

Generate a fresh signing secret for an endpoint. The previous secret is **retained** in the database (`secret_previous`) so that for the duration of the rotation, every delivery's `X-Fibo-Signature` header carries **both** signatures (`v1=<hash_with_new>` + `v1_prev=<hash_with_previous>`). Your verification handler can accept either, giving you a no-downtime deploy window.

```
POST /v1/webhook-endpoints/{id}/rotate-secret
```

Close the overlap with [`POST /v1/webhook-endpoints/{id}/finalize-rotation`](/docs/api-reference/webhook-endpoints/finalize-rotation) once the new secret is live on your server.

## Authentication

Requires `sk_*` for the same tenant, or admin token.

## Response

`201 Created`:

```json
{
  "id": "9a3e7d8b-…",
  "tenant_id": "00000000-0000-0000-0000-000000000001",
  "url": "https://your-app.example.com/fibo/webhooks",
  "secret": "<new 64-char hex secret>",
  "enabled_events": ["*"],
  "environment": "test",
  "rotation_pending": true,
  "updated_at": "2026-05-20T16:15:00.000Z"
}
```

:::warning Capture `secret` now
Same rule as create: the plaintext is shown ONCE. Subsequent reads on this endpoint only show `rotation_pending: true` until you call `finalize-rotation`.
:::

## Errors

| Status | Code | When |
|---|---|---|
| 400 | `invalid_request` | `id` is not a UUID. |
| 401 | `unauthenticated` | Missing / invalid / revoked bearer. |
| 403 | `forbidden` | You authenticated with a `pk_*`. |
| 404 | `not_found` | Endpoint doesn't exist, deleted, or in another tenant. |

## curl

```bash
curl -X POST https://api.fibo-crypto.fr/v1/webhook-endpoints/9a3e7d8b-…/rotate-secret \
  -H "Authorization: Bearer sk_test_..."
```

## Rotation flow

<details>
<summary>Step-by-step rotation without missing any events</summary>

1. **Mint** the new secret:
   ```bash
   curl -X POST .../webhook-endpoints/{id}/rotate-secret
   ```
   Response includes the new `secret`. Capture it.

2. **Deploy** the new secret to your server's environment. Your verifier should accept **either** the old or the new secret during the window:

   ```ts
   const valid =
     verifyHmac(req.body, sig, NEW_SECRET) ||
     verifyHmac(req.body, sig, OLD_SECRET);
   ```

   Every Fibo delivery during this window includes both `v1=<new>` and `v1_prev=<old>` in the `X-Fibo-Signature` header.

3. **Verify** the new secret is working — Fibo's deliveries during the window are signed with the new secret first, so if your server is using the new secret correctly the `v1` (not `v1_prev`) signature will validate.

4. **Finalize** — call [`POST /v1/webhook-endpoints/{id}/finalize-rotation`](/docs/api-reference/webhook-endpoints/finalize-rotation). After this point Fibo only includes `v1=<new>`. Your verifier should drop the old secret.

Don't skip step 4 — leaving both secrets valid indefinitely doubles your attack surface for as long as the old secret survives in any deployed environment.

</details>
